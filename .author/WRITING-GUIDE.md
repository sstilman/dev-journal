## 核心定位

### 四大板块一览

| 板块 | 对外定位 | 对内定位 | 回答的问题 | 写作心态 |
|------|----------|----------|------------|----------|
| **项目札记** | 开发过程记录 | 我在做什么 | "这个项目是怎么做的？" | 要讲清楚、有条理 |
| **技术随笔** | 技术知识沉淀 | 我学到了什么 | "这个技术是什么、怎么用？" | 要讲透彻、可查阅 |

| **博客** | 想法与感悟 | 我在想什么 | "我在想什么？" | 自由、随意、真实 |

## 五大板块详解

### 1. 项目札记 (docs/project-notes/)

**定位**：记录实际项目中的开发过程、技术调研、问题排查与解决方案。

**核心特征**：
- 围绕特定项目展开，有项目上下文依赖
- 记录"过程"：从零搭建 → 遇到问题 → 调试解决
- 有时间线：今天做了什么、明天要做什么
- 带有个人色彩：我的选择、我的踩坑、我的反思

**写作要点**：
- 开头交代项目背景和目标
- 详细记录遇到的问题和解决思路
- 适当穿插调试过程、命令输出、代码片段
- 结尾总结经验教训

**适合写的**：
- "我在这个项目中遇到了 XXX 问题，解决方案是..."
- "Dev Journal 博客系统搭建过程记录"
- "XX 项目技术选型与实现方案"

**不适合写的**：
- 通用的技术原理（放技术随笔）
- 从零开始的教学内容（放教程专栏）
- 与具体项目无关的工具推荐（放资源归档）

---

### 2. 技术随笔 (docs/tech-notes/)

**定位**：独立于项目的技术知识点深入讲解。

**核心特征**：
- 围绕特定技术展开，不绑定任何项目
- 讲解"原理"：概念 → 语法 → 应用场景 → 常见错误
- 独立成篇：任何项目、任何读者都能用
- 写成"文章"形式，而非流水账

**写作要点**：
- 先讲清楚技术是什么、解决什么问题
- 再讲解核心概念和语法
- 给出实用的代码示例
- 列出常见错误和使用注意事项
- 结尾给出延伸阅读和参考资料

**适合写的**：
- "TypeScript 泛型完全指南"
- "Git 工作流详解：从入门到精通"
- "Docker 网络配置实战"

**不适合写的**：
- 我在项目中遇到的具体问题（放项目札记）
- 保姆级入门教程（放教程专栏）
- 工具下载链接清单（放资源归档）

---

### 3. 资源归档 (docs/resources/)

**定位**：工具与资料索引清单。

**核心特征**：
- 以"清单"形式呈现，非长篇文章
- 不需要详细讲解，只需要说清楚是什么、怎么用
- 方便日后快速查阅
- 可以按主题、类型、用途等维度组织

**写作要点**：
- 每个条目包含：名称、简介、链接、适用场景
- 可以按评分、推荐程度排序
- 定期更新，及时移除过时内容
- 简洁明了，不需要废话

**适合写的**：
- "我常用的 VSCode 插件清单"
- "Linux 学习资料汇总"
- "免费图床服务对比"

---

### 4. 教程专栏 (tutorials/)

**定位**：系统性技术学习路径。

**核心特征**：
- 假设读者零基础，不过多依赖前置知识
- 按知识体系递进：从简单到复杂
- 每一步都有操作指引，可动手实践
- 配套练习或思考题巩固所学

**写作要点**：
- 先讲"是什么"和"为什么"，再讲"怎么做"
- 操作步骤清晰明确，带截图或代码
- 每个章节有小结和练习
- 避免过多术语堆砌，必要时解释术语
- 结构完整，像一本书的某一章

**适合写的**：
- "Linux 从入门到精通"系列
- "Node.js 后端开发实战教程"
- "React Hooks 完全指南"

**不适合写的**：
- 项目开发记录（放项目札记）
- 某个技术的个人理解（放技术随笔）
- 工具清单和使用方法（放资源归档）

---

### 5. 博客 (blog/)

**定位**：技术随想、阶段性总结、轻量分享。

**核心特征**：
- **最自由**：不要求结构完整，想到什么写什么
- **最真实**：可以有个人情绪，不必总是"客观中立"
- **最简短**：不是每篇都要长篇大论
- **最即时**：灵感来了就写，不要等到"准备充分"

**适合写的**：
- 💭 技术感悟：学完一个技术后的思考
- 📊 阶段总结：年终/季度复盘、项目总结
- 🛠️ 工具测评：一个工具好不好用的快速分享
- 🌱 成长记录：学习过程中的心路历程
- 📖 读书笔记：读完后记录核心观点和感悟
- 🚀 新技术尝鲜：对某个新技术的初步探索

## 写作原则

### 通用原则（适用于所有板块）

1. **原创优先**
   - 内容基于实践，避免纯翻译或复制
   - 可以引用他人的观点，但必须注明来源
   - 用自己的话重新组织，不要照搬

2. **可复现**
   - 操作步骤详细，读者能跟着做
   - 重要的命令、配置、代码完整呈现
   - 标明环境版本，避免歧义

3. **注明来源**
   - 参考资料的链接要清晰
   - 借鉴了哪些文章要标注
   - 方便读者延伸阅读

4. **持续更新**
   - 发现错误及时修正
   - 技术更新时补充新内容
   - 维护一个"最后更新时间"

### 各板块特殊原则

| 板块 | 特殊原则 |
|------|----------|
| 项目札记 | 保持时间线感，记录"今天做了什么" |
| 技术随笔 | 独立成篇，不依赖其他文档的前置知识 |
| 资源归档 | 定期清理，及时移除过时内容 |
| 教程专栏 | 假设读者零基础，不过度依赖前置知识 |
| 博客 | 自由真实，不要过度追求"完美" |

## Frontmatter 规范

### docs/ 文档

```yaml
---
id: unique-id                    # 唯一标识符（文件名作为默认 ID）
title: 文章标题                   # 可选，默认使用文件名
description: 简短描述             # SEO 和 hover card 显示
sidebar_position: 1              # 在侧边栏的排序位置
tags: [tag1, tag2]              # 标签（可选）
last_update:
  date: 2024-01-15
  author: Stilman
---
```

### blog/ 文章

```yaml
---
slug: custom-url-slug            # 自定义 URL（可选）
title: 文章标题                   # 可选，默认使用文件名
authors: stilman                 # 作者（需在 authors.yml 中定义）
tags: [tag1, tag2]              # 标签
draft: false                     # 草稿状态（true 不发布）
---
```

## 文件命名规范

### 命名格式

```
<板块>-<主题>-<简短描述>.md
```

### 命名示例

| 板块 | 示例文件名 |
|------|-----------|
| 项目札记 | `project-docusaurus-search-integration.md` |
| 技术随笔 | `tech-typescript-generics-guide.md` |
| 资源归档 | `resources-vscode-plugins-list.md` |
| 教程专栏 | `tutorial-linux-basics-commands.md` |
| 博客 | `2024-01-15-learning-reflections.md` |

### 命名规则

- 使用小写字母和连字符（`-`）
- 避免使用中文文件名（可能导致编码问题）
- 简短但有意义，一眼能看出内容主题
- 博客文章使用日期前缀便于排序

## 发布流程

### 草稿 → 发布

1. **写作**：使用对应板块的模板创建文件
2. **本地预览**：`yarn start`，访问 http://localhost:3000
3. **检查内容**：
   - [ ] 排版是否整齐
   - [ ] 代码块是否正确高亮
   - [ ] 图片是否正常显示
   - [ ] 链接是否有效
4. **发布**：
   - 提交到 GitHub
   - CI/CD 自动构建部署

### 定期维护

- 每季度检查一次资源归档，清理失效链接
- 发现错误及时修正，注明更新时间
- 回顾旧文章，更新过时的技术内容

## 常见问题

### Q: 这篇文章应该放项目札记还是技术随笔？

**判断标准**：
- 如果你正在做某个项目，遇到了问题并解决了 → **项目札记**
- 如果你想系统讲解某个技术，不绑定具体项目 → **技术随笔**

**例子**：
- ❌ 项目札记："TypeScript 泛型详解"（没有项目背景）
- ✅ 技术随笔："TypeScript 泛型详解"（系统讲解技术）
- ❌ 技术随笔："我在 Dev Journal 项目中配置搜索遇到的问题"（有项目背景）
- ✅ 项目札记："我在 Dev Journal 项目中配置搜索遇到的问题"（记录过程）
